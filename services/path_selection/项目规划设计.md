# Path Selection 优化方案

> 目标：在 **不修改 CodeQL 原始路径** 的前提下，稳定筛选出 3 条最符合目标 CVE 的数据流路径，并输出可追溯、可解释的报告。

---

## 设计原则
- **先过滤再打分**：把与 CVE 无关的噪声路径挡在体系外，避免后续环节被污染。
- **多源信息融合**：充分利用模块 1–5（CVE、Sink、Source、CodeQL 生成）以及 `services/knowledge_base` 中的结构化知识。
- **可解释 + 可回退**：每一步都保留证据与理由，LLM 只做增益，失败随时回到确定性结果。

---

## 处理流程

1. **CVE 背景解析**
   - 从 `output.md` 的 CVE Analysis、Sink/Source Analysis、情报摘要中抽取：
     - `vulnerability_type`
     - 预期 sink/source 描述、关键 API、危险函数
     - 明确提到的文件、包、目录
   - 合并模块 2–4 生成的 sink/source 列表和 CodeQL 模板上下文，构建 **文件/目录白名单** + **危险 API 集**。

2. **硬过滤**
   - 仅保留 sink 文件或目录落在白名单范围内的路径。
   - 进一步检查：
     - sink 描述/代码片段是否包含 CVE 关键词或危险 API；
     - source 是否来自用户输入相关文件/函数；完全不符直接 PASS；
     - threadFlow 是否完整，排除 test/初始化等明显噪声。

3. **特征提取（只读）**
   - 遍历每条 `threadFlow` 的所有步骤，记录：
     - source/sink 文件、行号、函数、API
     - 中间步骤的危险 API、潜在 sanitization、跨文件/跨函数情况
     - 路径长度、跳步、是否经过认证/过滤模块
     - 命中的 CVE/知识库标签（如 SSRF → HTTP 请求）
   - 结果写入 `PathFeatures` 数据类，连同 `original_path` 一并交给后续阶段。

4. **确定性打分 + 覆盖约束**
   - 对每条路径计算 5 个指标 `m_i`（0–1）：
     - `m_file`：sink 文件/目录匹配程度（精确=1，模糊=0.6，否则 0）
     - `m_sink`：危险 API、知识库标签、函数签名的匹配程度
     - `m_src`：source 是否由用户输入驱动且未被净化
     - `m_flow`：路径结构质量（长度、连续性、跨界合理性、无 source=sink）
     - `m_cov`：对 CVE 关键点（sink/source 组合、危险 API）的覆盖贡献
   - 权重 `w_i` 由 `vulnerability_type` + 知识库配置驱动（如 SSRF 强化 `w_sink`，RCE 提升 `w_src`）。计算确定性得分：
     ```
     score_det = Σ (w_i · m_i)
     ```
   - 按得分排序，并执行覆盖约束：
     1. 先选择 sink 文件互异的路径；
     2. 若不足 `top_k`，补充 source 类型互异的路径；
     3. 仍不足则按得分继续放宽。
   - 这一阶段即便 LLM 离线，也能保证有 3 条高质量候选。

5. **LLM 点读 + 语义精排**
  > 明确把 “只允许在候选内选 3 条”、“不得生成新路径/改动路径顺序/行号” 写进 system prompt,防止AI乱生成或者作弊

   - 使用 `CodeContextBuilder` 对每条候选的 source、sink 及关键中间步骤做“点读”：读取各自前后 5–8 行，形成结构化片段 `{"role": "source", "file": "...", "snippet": "..."}`，保证上下文短小、按源→中间→汇排列，并附带危险 API / 函数元信息。
   - 将前 N（默认 6）条候选的特征摘要 + 点读片段喂给 LLM，明确要求它只在这些候选中挑选最终 3 条，并输出结构化 JSON：
     ```json
     {
       "path_index": 2,
       "llm_alignment_score": 0.84,
       "source_match": true,
       "sink_match": true,
       "notes": "request.body 拼接到 httpx.get(URL)，满足 SSRF 场景"
     }
     ```
   - 返回体必须通过 `pydantic` schema 校验；若 LLM 失败或违反约束，立即回退到纯确定性结果。
   - 融合得分时使用 `score_final = 0.7 * score_det + 0.3 * llm_alignment_score`（权重可按需要调整），LLM 只负责语义加分和理由解释。

6. **规则化验证**
   - `PathVerifier` 基于 `PathFeatures` + 得分结果执行硬规则与警告：
     - **硬规则**：`m_sink < 0.6`、`m_src < 0.5`、缺少危险 API、路径长度 <3、source/sink 行号相同等 → 判 invalid。
     - **警告**：LLM 对齐分过低、缺少代码上下文、未覆盖全部关键危险函数等。
   - 若某条候选 invalid，则按得分从剩余路径中自动补位，直到满足 `top_k` 或耗尽候选。

7. **报告输出**
    最后输出最终选择的三条路径的“标准结果”json格式,示例结构如下
   ```
   {
  "dataFlowPath": [
    {
      "threadFlows": [
        {
          "steps": [

              }
            },
            {
              "stepNumber": 2,

              }
   ```

## 实现建议
1. **特征层**：新增 `PathFeatures` + `PathFeatureExtractor`，取代 `PathEnricher`。
2. **权重配置**：在 `services/knowledge_base` 中维护漏洞类型 → (`w_i`, 危险 API, sanitizers) 映射，供 scorer 读取。
3. **打分与排序**：实现 `PathRanker`，负责硬过滤、特征打分、覆盖约束，并在 `selector.py` 中先于 LLM 调用。
4. **点读 & LLM**：实现 `CodeContextBuilder` + LLM schema 校验，将 `llm_alignment_score` 融入最终得分并支持回退。
5. **验证器升级**：扩展 `PathVerifier` 以引用特征/得分，执行硬规则判定并自动补位。
6. **测试**：为特征提取、ranker、LLM schema、verifier 分别编写单测，确保第 6 模块成为真正的质量闸口。

通过“硬过滤 → 确定性打分 → 点读语义精排 → 规则验证”这一链路，路径选择既能过滤掉落在错误文件的噪声，又能利用 LLM 对代码语义的理解，输出透明、可追溯的结果。
