# 设计文档：Sink和Source Agent集成CodeQLComposeTool

## 上下文
当前Sink和Source查找Agent使用直接的文件文本分析来识别潜在的危险函数和数据源点。这种方式虽然简单，但缺乏语义理解能力，容易产生误报和漏报。

## 目标/非目标
- **目标**: 将Sink和Source Agent集成到CodeQLComposeTool，利用CodeQL的语义分析能力
- **目标**: 保持现有接口和输出格式100%兼容
- **非目标**: 修改CodeQLComposeTool核心功能
- **非目标**: 改变Agent的公共接口

## 技术决策

### 决策1: 使用exec_mode='run'模式
**原因**: 
- Sink和Source查找需要文本格式结果便于分析
- `exec_mode='run'`提供文本输出和文件路径
- 避免复杂的SARIF解析逻辑

**替代方案考虑**:
- `exec_mode='analyze'` + SARIF解析：更复杂，需要额外的SARIF处理逻辑
- 直接调用codeql命令：缺乏错误处理和迭代优化能力

### 决策2: 提示词驱动的集成方式
**原因**:
- 最小化代码变更
- 保持Agent接口不变
- 通过提示词控制行为，灵活性高

**替代方案考虑**:
- 修改Agent继承结构：变更过大，影响现有代码
- 创建新的包装类：增加不必要的复杂性

### 决策3: 保持回退机制
**原因**:
- 确保系统稳定性
- CodeQL查询可能生成失败
- 向后兼容性要求

## 风险/权衡
- **性能风险**: CodeQL查询执行比直接文件分析慢
  - **缓解**: 通过优化提示词减少迭代次数，缓存常用查询
- **精度风险**: CodeQL查询可能无法覆盖所有复杂场景
  - **缓解**: 保留文件工具作为回退，结合两种方法的优势
- **兼容性风险**: 新的集成方式可能影响现有流程
  - **缓解**: 保持接口不变，通过配置开关控制新功能

## 迁移计划
1. **阶段1**: 更新提示词模板，添加CodeQLComposeTool使用指导
2. **阶段2**: 在Agent中添加CodeQLComposeTool调用逻辑
3. **阶段3**: 实现SARIF结果解析和报告生成
4. **阶段4**: 添加错误处理和回退机制
5. **阶段5**: 全面测试和性能优化
