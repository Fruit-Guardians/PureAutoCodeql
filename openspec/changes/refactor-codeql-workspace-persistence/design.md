# 设计文档：CodeQL 工作区持久化策略重构

## Context
当前系统在每次生成 CodeQL 查询时，都会在 `temp/codeql_temp` 下创建一个基于时间戳的新文件夹（如 `20251111_123456_789012`）。这种设计导致：
1. 临时文件夹堆积问题
2. 错误修复流程效率低下（每次都重新生成而非修改）
3. 难以追踪同一任务的多轮迭代历史

本次重构目标是实现任务级别的工作区管理，支持文件原地修改。

## Goals / Non-Goals

### Goals
- 每个任务使用唯一且固定的工作区路径 `temp/codeql_temp/<task_id>`
- 首轮生成后，将查询保存到持久化目录 `temp/ql_queries/<task_id>`
- 错误修复阶段使用 MCP 文件系统工具原地修改文件，而非重新生成
- 保持向后兼容性，未指定 `task_id` 时仍使用时间戳模式

### Non-Goals
- 不重构整个 CodeQL 执行流程
- 不改变 SARIF 输出的目录结构（`output/result_*.sarif`）
- 不实现任务间的工作区共享或缓存机制
- 不自动清理旧的时间戳目录（可由后续清理任务处理）

## Decisions

### 决策 1：任务 ID 的来源与格式
- **选择**：由调用方（`CodeQLComposeTool`）生成和管理任务 ID
- **格式**：使用时间戳 + 随机后缀（如 `20251111_123456_abc123`）或 UUID
- **理由**：
  - 保持 `codeql.py` 工具层的简单性，不引入复杂的 ID 生成逻辑
  - 调用方已有上下文信息，更适合管理任务生命周期
  - 便于在 API 层传递和追踪任务状态

### 决策 2：工作区复用策略
- **选择**：如果工作区目录已存在，直接复用（覆盖文件）
- **理由**：
  - 支持任务重试场景（如进程崩溃后重启）
  - 避免手动清理导致的失败
  - 简化实现逻辑
- **备选方案**：清空目录后重新创建
  - 优点：确保环境干净
  - 缺点：可能意外删除重要文件

### 决策 3：持久化保存时机
- **选择**：在 `CodeQLComposeTool` 首次语法校验通过后保存
- **理由**：
  - 语法校验通过表示查询可编译，值得保存
  - 执行结果（有无漏洞）不影响查询本身的有效性
  - 便于后续人工审查或复用
- **备选方案**：在执行成功后保存
  - 缺点：空结果查询不会被保存，但仍可能有价值

### 决策 4：原地修复的实现方式
- **选择**：通过新 prompt 指导 LLM 调用 `@modelcontextprotocol/server-filesystem` 工具
- **理由**：
  - 充分利用现有 MCP 工具生态，无需自定义文件操作工具
  - LLM 对正则替换有良好理解能力
  - 保持代理层的简洁性，不引入复杂的文件操作逻辑
- **备选方案**：在 Python 代码中解析修复建议并应用
  - 缺点：需要复杂的 AST 解析或 diff 应用逻辑

### 决策 5：错误修复模式切换时机（已更新）
- **选择**：首轮生成后立即保存到文件，后续所有修复都使用原地修改模式
- **理由**：
  - 无需等待"首次成功"，首轮生成后就有 .ql 文件可供修改
  - 统一的修复流程，简化逻辑，无需判断是否首次成功
  - 充分利用 MCP 工具的文件修改能力
  - 保留完整的修改历史（文件逐步改进）
- **备选方案**：首次成功后才切换到原地修改
  - 缺点：增加复杂度，需要额外的状态追踪

### 决策 6：Agent 职责拆分
- **选择**：将原来的 `CodeQLErrorAgent` 两种模式拆分为两个独立 Agent
  - `CodeQLErrorAgent`：只负责错误分析，生成修复建议
  - `CodeQLFixInplaceAgent`：只负责原地修复，使用 MCP 工具修改文件
- **理由**：
  - 单一职责原则，每个 Agent 只做一件事
  - 更清晰的代码结构和更好的可测试性
  - 灵活的组合方式，可独立替换或优化
  - 避免在一个类中混杂两种不同的行为模式
- **备选方案**：保留 `fix_mode` 参数在一个 Agent 中
  - 缺点：违反单一职责原则，代码复杂度高

## Risks / Trade-offs

### 风险 1：并发任务冲突
- **描述**：如果两个任务使用相同的 `task_id`，会互相覆盖文件
- **缓解措施**：
  - 调用方确保生成唯一 `task_id`（使用 UUID 或加锁）
  - 添加文件锁机制（未来优化）

### 风险 2：MCP 工具调用失败
- **描述**：LLM 可能无法正确使用 `@modelcontextprotocol/server-filesystem` 工具
- **缓解措施**：
  - Prompt 中提供详细的工具使用示例
  - 如果 `inplace` 模式失败，回退到 `generate` 模式
  - 在 `CodeQLComposeTool` 中检测工具调用错误并重试

### 风险 3：向后兼容性
- **描述**：现有代码路径可能未传递 `task_id`，导致行为变化
- **缓解措施**：
  - 默认参数 `task_id=None`，保持时间戳模式
  - 逐步迁移现有调用点
  - 添加单元测试覆盖两种模式

### Trade-off：磁盘空间 vs. 追踪能力
- **当前设计**：工作区保留在 `temp/` 中，需要手动清理
- **优点**：便于调试，可查看中间状态
- **缺点**：长期运行会占用磁盘空间
- **未来优化**：添加定期清理任务或基于 LRU 策略

## Migration Plan

### Phase 1：基础设施（不影响现有功能）
1. 修改 `create_temporary_qlpack` 添加可选 `task_id` 参数
2. 添加持久化保存函数 `save_query_to_persistent_dir`
3. 创建 `codeql_fix_inplace.md` prompt
4. 添加单元测试

### Phase 2：集成到主流程
1. 修改 `CodeQLComposeTool` 生成并传递 `task_id`
2. 首轮生成后立即保存到工作区文件和持久化目录
3. 拆分 Agent：`CodeQLErrorAgent`（错误分析）+ `CodeQLFixInplaceAgent`（原地修复）
4. 所有修复轮次使用原地修改模式
5. 集成测试验证端到端流程

### Phase 3：逐步迁移现有调用
1. 更新 API 层传递 `task_id`
2. 更新其他调用 `execute_codeql_query` 的地方
3. 监控并修复潜在问题

### Rollback 策略
- 如果发现严重问题，将 `task_id` 默认值改为生成时间戳，回退到旧行为
- 保留旧的时间戳逻辑代码路径，确保快速回退
- 持久化目录是附加功能，不影响核心流程

## Open Questions

1. **任务 ID 格式标准化**：是否需要定义统一的 ID 格式规范？
   - 建议：使用 `{timestamp}_{uuid4}`，便于排序和唯一性
2. **持久化目录的版本管理**：是否需要保存多个版本的查询？
   - 建议：初期仅保存最新成功版本，未来可扩展为版本历史
3. **跨会话任务恢复**：是否支持进程重启后继续任务？
   - 建议：初期不支持，需要额外的状态持久化机制
4. **原地修复失败后的回退策略**：是否自动切换回生成模式？
   - 建议：初期手动回退，收集数据后优化策略
