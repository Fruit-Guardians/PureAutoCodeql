# 路径选择与结果质量检测技术文档

## 1. 概述

路径选择与结果质量检测模块是 PureAutoCodeQL 分析流水线的第六个核心组件，负责从 CodeQL 查询返回的大量数据流路径中智能筛选出最符合目标 CVE 漏洞特征的路径，并通过多维度验证确保选中路径的质量。

### 1.1 设计目标

- **精准筛选**：从数十甚至上百条 CodeQL 路径中选出最相关的 Top-K 条路径
- **语义理解**：基于 LLM 理解 CVE 真实含义，而非僵硬的规则匹配
- **质量保证**：通过多层验证机制确保选中路径的完整性、正确性和置信度
- **可解释性**：为每条选中的路径提供清晰的选择理由和分析步骤
- **鲁棒性**：LLM 失败时自动降级到确定性打分方案

### 1.2 核心价值

在 CodeQL 漏洞检测中，一次查询可能返回几十甚至上百条数据流路径，但其中：
- 大部分路径是测试代码、示例代码或无关的正常业务流程
- 真正与 CVE 漏洞相关的路径可能只有 2-3 条
- 人工筛选耗时且容易遗漏关键路径

本模块通过**确定性打分 + LLM语义理解 + 多维验证**的三级流水线，实现了自动化的高质量路径筛选。

### 2.2 六步处理链路

1. **CVE 背景解析** - 从 `output.md` 提取 CVE 关键信息
2. **特征提取** - 为每条路径提取结构化特征（只读，不修改路径）
3. **确定性打分** - 基于规则和特征计算每条路径的得分
4. **LLM 点读精排** - 利用 LLM 进行语义理解和路径排序
5. **多维验证** - 验证选中路径的完整性、正确性和置信度
6. **报告输出** - 生成 Markdown、JSON 和 dataFlowPath 导出

**位置**: `services/path_selection/context_extractor.py`

**职责**: 从前序模块生成的 `output.md` 中提取 CVE 相关的关键信息。

**提取内容**:

{

  "vulnerability_type": "SSRF",
  "expected_sink": "httpx.get, requests.get",
  "expected_source": "request.body, request.params",
  "technical_details": "用户输入未经验证直接传入HTTP请求...",
  "sink_analysis": {...},  # 来自 Sink 分析模块
  "source_analysis": {...} # 来自 Source 分析模块

```

**关键特性**:
- 充分利用前面 Pipeline 步骤（CVE 分析、Sink 分析、Source 分析）的结果
- 构建**文件/目录白名单**和**危险 API 集合**
- 为后续打分和 LLM 分析提供上下文依据

### 3.2 路径特征提取器 (PathFeatureExtractor)

**位置**: `services/path_selection/path_feature_extractor.py`

**职责**: 为每条 CodeQL 路径提取结构化特征，但**不修改原始路径数据**。

#### 3.2.1 特征结构

```python
@dataclass
class PathFeatures:
    index: int                      # 路径索引
    original_path: Dict[str, Any]   # 原始 threadFlows 数据
    path_length: int                # 路径步数
    
    # 关键节点
    source: PathNode                # Source 端点
    sink: PathNode                  # Sink 端点
    intermediates: List[PathNode]   # 中间步骤节点
    
    # 特征信息
    dangerous_apis: List[str]       # 命中的危险 API
    sanitizer_hits: List[str]       # 命中的净化函数
    matched_keywords: List[str]     # 匹配的 CVE 关键词
    
    # 元数据
    language: str
    metadata: Dict[str, Any]
    flow_summary: str              # 自动生成的流程摘要
```

#### 3.2.2 PathNode 结构

每个节点（Source/Sink/Intermediate）都包含：

```python
@dataclass
class PathNode:
    role: str                      # source / sink / intermediate_N
    file: str                      # 文件路径
    line: int                      # 起始行号
    end_line: int | None          # 结束行号
    description: str               # CodeQL 节点描述
    code_snippet: str              # 代码片段（带行号和标记）
    analysis: Dict[str, Any]       # 语言适配器分析结果
    api_calls: List[str]           # 提取的 API 调用
```

#### 3.2.3 代码片段读取

特征提取器会为每个关键节点读取**前后 5 行代码**：

```
>>> 164 | def parse_request(self, request):
    165 |     content = request.body()
    166 |     data = json.loads(content)
    167 |     url = data.get('url')
    168 |     return httpx.get(url)
```

- 使用 `>>>` 标记关键行
- 包含行号便于定位
- 支持多层目录搜索和路径缓存优化

#### 3.2.4 语言适配器

通过 `LanguageAdapter` 接口支持多语言分析：

```python
# Python 适配器
- Source 识别: request, param, input, body, query
- Sink 识别: exec, eval, system, httpx.get, pickle.loads

# Java 适配器  
- Source 识别: HttpServletRequest, @RequestParam
- Sink 识别: Runtime.exec, ProcessBuilder, JNDI.lookup

# C/C++ 适配器
- Source 识别: argv, stdin, gets, recv
- Sink 识别: system, strcpy, sprintf, memcpy
```

### 3.3 确定性打分器 (PathRanker)

**位置**: `services/path_selection/path_ranker.py`

**职责**: 基于规则和特征为每条路径计算确定性得分，并执行覆盖约束选择候选路径。

#### 3.3.1 五维评分指标

每条路径计算 5 个维度的分数 (0-1)：

| 维度 | 权重 | 计算逻辑 |
|------|------|---------|
| **m_file** | 0.25 | Sink 文件是否匹配 CVE 提及的文件/目录 |
| **m_sink** | 0.25 | 是否命中危险 API、匹配预期 Sink 描述 |
| **m_source** | 0.20 | Source 是否来自用户输入、类型是否明确 |
| **m_flow** | 0.15 | 路径长度合理性 (4-8步最优)、跨文件流动 |
| **m_coverage** | 0.15 | 匹配 CVE 关键词的覆盖率 |

**总分计算**:
```python
score_det = Σ (w_i · m_i)
```

#### 3.3.2 动态权重调整

根据漏洞类型自动调整权重：

```python
PROFILE_OVERRIDES = {
    "ssrf": {
        "sink": 0.35,      # SSRF 重点关注 Sink
        "file": 0.25,
        "source": 0.2,
        "flow": 0.1,
        "coverage": 0.1
    },
    "rce": {
        "sink": 0.3,       # RCE 同时重视 Source 和 Sink
        "source": 0.3,
        "file": 0.2,
        "flow": 0.1,
        "coverage": 0.1
    },
    "deserialization": {
        "sink": 0.3,       # 反序列化重点关注特定文件
        "file": 0.25
    }
}
```

#### 3.3.3 三阶段候选选择

```python
# Phase 1: 确保 Sink 文件多样性
for score in ordered:
    sink_file = score.feature.sink.file
    if sink_file not in used_sinks:
        selected.append(score)
        used_sinks.add(sink_file)
    if len(selected) >= top_k:
        break

# Phase 2: 确保 Source 类型多样性
for score in ordered:
    source_type = score.feature.source.analysis.get("type")
    if source_type not in used_source_types:
        selected.append(score)
        used_source_types.add(source_type)

# Phase 3: 按得分填充剩余槽位
for score in ordered:
    if score.feature.index not in used_indices:
        selected.append(score)
```

这种分阶段选择确保了：
- **多样性**：避免选择的路径过于相似
- **覆盖率**：尽可能覆盖不同的 Sink 文件和 Source 类型
- **质量**：在多样性基础上优先选择高分路径

### 3.4 LLM 语义分析器 (LLMPathAnalyzer)

**位置**: `services/path_selection/llm_analyzer.py`

**职责**: 利用大语言模型对候选路径进行语义理解和精确排序。

#### 3.4.1 点读上下文构建

为每条候选路径构建**结构化代码片段**：

```python
# CodeContextBuilder 输出格式
{
  "role": "source",
  "file": "src/app.py",
  "location": "src/app.py:164",
  "description": "ControlFlowNode for Attribute",
  "snippet": """
    >>> 164 | def parse_request(self, request):
        165 |     content = request.body()
        166 |     data = json.loads(content)
  """
}
```

关键设计：
- **有序排列**: Source → Intermediates → Sink
- **限制中间节点**: 最多保留 3 个中间步骤，避免上下文过长
- **截断保护**: 每个代码片段最长 1200 字符

#### 3.4.2 LLM Prompt 结构

```markdown
# CVE 信息
- CVE ID: CVE-2025-54381
- 漏洞类型: SSRF
- 预期的 Sink: httpx.get, requests.get
- 预期的 Source: request.body

# 候选路径
## 候选 0 (path_index=2, det_score=0.847)
- 流程摘要: request.body → json.loads → httpx.get
- 关键API: httpx.get
- 确信原因: 命中危险API, Sink文件匹配

### SOURCE (src/app.py:164)
    >>> 164 | def parse_request(self, request):
        165 |     content = request.body()

### SINK (src/app.py:169)
    >>> 169 |     return httpx.get(url)

# 指令
选择 **3** 个最相关的候选路径。必须引用上方点读块中的证据。

返回 JSON 格式:
{
  "selected_paths": [
    {
      "candidate_rank": 0,
      "llm_alignment_score": 0.85,
      "reason": "从 request.body 到 httpx.get 的路径匹配 SSRF",
      "analysis_steps": [
        "确认 source 来自 request.body",
        "路径经过 decode 后直接进入 httpx.get",
        "未发现有效过滤，符合 SSRF"
      ],
      "evidence": [
        {
          "block_role": "source",
          "location": "src/app.py:164",
          "insight": "request.body() 直接读取用户输入"
        }
      ]
    }
  ],
  "overall_reasoning": "...",
  "coverage_analysis": {...}
}
```

#### 3.4.3 响应解析与融合

```python
# 1. Pydantic 模型校验
parsed = AnalyzerResponseModel.model_validate_json(json_str)

# 2. 融合确定性得分和 LLM 得分
blended = 0.7 * deterministic_score + 0.3 * llm_alignment_score

# 3. 回写到 selection_info
selection_info.update({
    "llm_alignment_score": 0.85,
    "reason": "从 request.body 到 httpx.get...",
    "analysis_steps": [...],
    "evidence": [...],
    "confidence": round(blended, 4)
})
```

**关键约束**:
- LLM **只能从候选中选择**，不得生成新路径
- 必须输出结构化 JSON，通过 Pydantic 严格校验
- 失败时自动降级到确定性打分结果

#### 3.4.4 流式输出支持

LLMPathAnalyzer 支持**流式输出**，在分析过程中实时展示：

```python
async def astream(prompt):
    async for chunk in llm.astream(prompt):
        # 实时推送给前端或控制台
        await event_callback({
            "type": "agent_thinking",
            "message": chunk.content,
            "data": {"is_final": False}
        })
```

优化控制台输出：
- 流式输出时只显示进度点 (`.`)
- 最终输出时美化显示 JSON 摘要
- 支持 SSE 推送到前端界面

### 3.5 路径验证器 (PathVerifier)

**位置**: `services/path_selection/path_verifier.py`

**职责**: 对 LLM 选中的路径进行四重验证。

#### 3.5.1 四维验证体系

```python
verification = {
    "is_valid": bool,              # 总体是否有效
    "issues": List[str],           # 阻断性问题
    "warnings": List[str],         # 非阻断性警告
    "completeness": {...},         # 完整性检查
    "correctness": {...},          # 正确性检查
    "semantics": {...},            # 语义性检查
    "confidence": {...}            # 置信度检查
}
```

#### 3.5.2 完整性检查 (Completeness)

验证路径的结构完整性：

```python
def _check_completeness(path):
    issues = []
    
    # 1. Source/Sink 位置信息完整
    if not path.get("source_location"):
        issues.append("缺少Source位置信息")
    if not path.get("sink_location"):
        issues.append("缺少Sink位置信息")
    
    # 2. 路径长度合理
    if path.get("path_length", 0) < 2:
        issues.append(f"路径太短 (只有{path_length}步)")
    
    # 3. 文件和行号存在
    for label, loc in [("Source", source_loc), ("Sink", sink_loc)]:
        if not loc.get("file"):
            issues.append(f"{label}缺少文件信息")
        if not loc.get("startLine"):
            issues.append(f"{label}缺少行号信息")
    
    return {"valid": len(issues) == 0, "issues": issues}
```

#### 3.5.3 正确性检查 (Correctness)

验证路径是否匹配 CVE 预期：

```python
def _check_correctness(path, cve_context):
    warnings = []
    
    expected_sink = cve_context.get("expected_sink", "")
    expected_source = cve_context.get("expected_source", "")
    
    # 1. Sink 匹配度（降级为警告）
    if expected_sink:
        if not any(token in sink_desc for token in expected_sink.split()):
            warnings.append("Sink未完全匹配CVE预期目标")
    
    # 2. Source 合理性（降级为警告）
    if expected_source:
        if not any(token in source_desc for token in expected_source.split()):
            warnings.append("Source未显示预期相关点")
    
    return {"valid": True, "warnings": warnings}
```

**设计考量**: 将大部分检查降级为 `warnings` 而非 `issues`，避免过于严格导致误杀。

#### 3.5.4 语义检查 (Semantics)

验证路径的语义合理性：

```python
def _check_semantics(path, cve_context):
    warnings = []
    
    # 1. 危险 API 存在性
    if not path.get("dangerous_apis"):
        warnings.append("未显示明显的危险API")
    
    # 2. CVE 关键词匹配
    matched_keywords = path.get("matched_keywords", [])
    cve_keywords = extract_keywords(cve_context.get("technical_details"))
    if cve_keywords and not matched_keywords:
        warnings.append("路径未包含CVE相关关键词")
    
    # 3. Source 类型识别
    source_type = path.get("source_analysis", {}).get("type")
    if source_type in ("unknown", "", None):
        warnings.append("Source类型未明确标记为用户输入")
    
    return {"valid": True, "warnings": warnings}
```

#### 3.5.5 置信度检查 (Confidence)

综合评估路径的可信度：

```python
def _check_confidence(path):
    deterministic = path.get("selection_info", {}).get("deterministic_score", 0.0)
    llm_score = path.get("selection_info", {}).get("llm_alignment_score", 0.0)
    
    # 融合得分
    blended = 0.7 * deterministic + 0.3 * llm_score
    
    # 阈值判断
    if blended < 0.5:
        issues.append(f"置信度过低 ({blended:.2f})")
    elif blended < 0.7:
        warnings.append(f"置信度较低 ({blended:.2f})")
    
    return {
        "confidence_score": round(blended, 4),
        "valid": blended >= 0.5
    }
```

### 3.6 断流检测与补边 (FlowBreakDetection)

**位置**: `services/flow_break_detection.py`

**职责**: 检测 CodeQL 数据流分析中的断流点，并自动生成 `isAdditionalFlowStep` 子句进行补边。

#### 3.6.1 什么是断流？

CodeQL 的 DataFlow 分析可能在某些情况下无法自动识别数据流传递：

```python
# 示例：通过字典传递数据
def process_request(request):
    data = {"url": request.body()}  # Source
    params = data["url"]             # ❌ CodeQL 可能断流
    return httpx.get(params)         # Sink
```

在这种情况下，CodeQL 可能无法识别 `data["url"]` 到 `params` 的流动，导致漏报。

#### 3.6.2 断流检测原理

通过构造特殊的双向查询来定位断流点：

```ql
// 前向查询：从 fixed source 出发能到达的节点
module ForwardCfg {
  predicate isSource(Node src) { src instanceof FixedSourceNode }
  predicate isSink(Node sink) { anyNode(sink) }
}

// "后向"查询：能流向 fixed sink 的节点
module BackwardCfg {
  predicate isSource(Node src) { anyNode(src) }
  predicate isSink(Node sink) { sink instanceof FixedSinkNode }
}

// 断流候选：同时满足前向可达和后向可达，但中间断开
from Node n
where forwardReachable(n) and backwardReachable(n)
select n, "Flow break candidate"
```

#### 3.6.3 自动补边生成

对检测到的断流点，自动生成补边子句：

```ql
predicate isAdditionalFlowStep(DataFlow::Node src, DataFlow::Node dst) {
  // 原有补边逻辑
  (
    ... existing logic ...
  )
  or
  // 新增补边 (针对 serde.py:169 列6)
  exists(
    DataFlow::Node __fb_src,
    DataFlow::Node __fb_dst |
    FlowBreakSupport::connectSameLine("src/serde.py", 169, 6, __fb_src, __fb_dst)
  | src = __fb_src and dst = __fb_dst
  )
}
```

#### 3.6.4 多轮迭代策略

```python
config = FlowBreakDetectionConfig(
    enable=True,
    max_rounds=3,                    # 最多3轮迭代
    max_clauses_per_round=5,         # 每轮最多添加5个子句
    max_total_clauses=12,            # 总计最多12个子句
    candidate_soft_limit=200,        # 候选数超过200时截断
    candidate_cap=50                 # 截断后保留50个候选
)
```

每轮迭代：
1. 执行断流检测查询
2. 解析 SARIF 结果获取候选断流点
3. 生成 `isAdditionalFlowStep` 子句
4. 合并到原始查询中
5. 重新执行主查询，检查是否产生新路径

### 4.2 内部数据流

```python
# 1. 加载原始路径

## 5. 关键技术亮点

### 5.1 点读 (Code Context) 技术

**核心思想**: 不向 LLM 发送完整代码文件，而是针对关键节点"点读"代码片段。

**优势**:
- **上下文精简**: 每条候选路径的上下文控制在 2-3KB
- **关注重点**: LLM 只看到 Source/Sink/关键中间步骤的代码
- **成本优化**: 大幅减少 token 消耗，提升响应速度

**实现细节**:
```python
# 为每个节点读取前后 5 行代码
snippet = read_lines(file, line - 5, line + 5)

# 标记关键行
for idx, line in enumerate(lines):
    prefix = ">>> " if idx + 1 == target_line else "    "
    output += f"{prefix}{idx+1:4d} | {line}"
```

### 5.2 确定性得分作为 LLM 输入

**设计理念**: LLM 不应从零开始评估，而应在确定性得分基础上进行精排。

**流程**:
1. 确定性打分筛选出 Top-6 候选
2. 将候选的得分和理由发送给 LLM
3. LLM 基于代码语义进行微调和重排序

**好处**:
- 减少 LLM 的搜索空间（从 100+ 到 6）
- 为 LLM 提供参考依据，提升判断准确性
- LLM 失败时可直接使用确定性结果

### 5.3 融合得分机制

```python
confidence = 0.7 * deterministic_score + 0.3 * llm_alignment_score
```

**权重设计考量**:
- **70% 确定性得分**: 确保基础质量，避免 LLM 幻觉
- **30% LLM 得分**: 利用语义理解优势，提升精确度

**动态调整策略** (可扩展):
```python
if llm_confidence_high:
    # LLM 非常确定时提升其权重
    confidence = 0.5 * det + 0.5 * llm
elif llm_uncertainty:
    # LLM 不确定时降低其权重
    confidence = 0.85 * det + 0.15 * llm
```

### 5.4 异步并发优化

**特征提取阶段**使用异步 I/O 加速代码读取：

```python
async def build_features(paths, source_root):
    tasks = [
        build_single_feature(idx, path, source_root)
        for idx, path in enumerate(paths)
    ]
    # 并发处理 100+ 条路径
    features = await asyncio.gather(*tasks)
    return features

async def _read_code_snippet(source_root, file, line):
    # 将文件读取放到线程池执行，避免阻塞事件循环
    return await asyncio.to_thread(
        _sync_read_code_snippet, source_root, file, line
    )
```

**性能提升**:
- 100 条路径的特征提取从 **10-15 秒** 降低到 **2-3 秒**
- 充分利用多核 CPU 和 I/O 并发

### 5.5 多层降级策略

```python
try:
    # 第一优先：LLM 流式输出
    result = await llm.astream(prompt)
except StreamError:
    try:
        # 第二优先：LLM 普通调用
        result = await llm.ainvoke(prompt)
    except LLMError:
        # 最终降级：确定性打分结果
        result = fallback_to_deterministic_scores()
```

**降级层次**:
1. LLM 流式输出（最佳体验）
2. LLM 普通调用（确保功能）
3. 确定性打分（保底方案）

## 11. 故障排查

### 11.1 常见问题

#### Q1: LLM 返回的 JSON 格式不正确

**现象**: `ValidationError: JSON parsing failed`

**原因**: LLM 输出了非标准 JSON 或包含额外的 Markdown 语法

**解决**:
```python
# 系统会自动尝试清理 JSON
def _extract_json(text):
    text = text.strip()
    if text.startswith("```json"):
        text = text[7:]
    elif text.startswith("```"):
        text = text[3:]
    if text.endswith("```"):
        text = text[:-3]
    return text.strip()
```

如果仍然失败，会自动降级到确定性打分结果。

#### Q2: 代码片段无法读取

**现象**: `snippet: _未获取代码片段_`

**可能原因**:
1. `source_root` 路径不正确
2. 相对路径解析失败
3. 文件权限问题

**排查步骤**:
```python
# 1. 检查 source_root 是否存在
assert Path(source_root).exists()

logging.getLogger("services.path_selection").setLevel(logging.DEBUG)
```

#### Q3: 确定性得分全为 0

**现象**: 所有路径的 `deterministic_score` 都是 0

class GoLanguageAdapter(LanguageAdapter):

class CustomPathRanker(PathRanker):
    """自定义打分策略"""
    
    def _weights_for_context(self, cve_context):
        weights = super()._weights_for_context(cve_context)
        
        # 对于特定项目，强化文件匹配权重
        if cve_context.get("project") == "critical_system":
            weights["file"] = 0.4
            weights["sink"] = 0.3

        if "auth" in feature.sink.file.lower():

            reason += "; 位于认证模块"
        
        return min(score, 1.0), reason

# 使用自定义打分器
service.path_ranker = CustomPathRanker(language="python")
