# 代码层面的源码分析增强方案 🔒

## 问题：为什么需要代码层面的限制？

### 之前的方案（只靠提示词）
```
提示词告诉LLM: "请使用server-filesystem读取源文件"
    ↓
LLM可能会读 ✅
    ↓
LLM可能不读 ❌ (如果LLM忽略指令)
```

**风险**：完全依赖LLM的"自觉性"

### 增强方案（代码主动控制）
```
代码主动读取源文件
    ↓
直接注入到提示词中
    ↓
LLM必然看到源码 ✅ (代码保证)
```

**优势**：不依赖LLM行为，代码强制保障

**新增方法**：

#### 1. `_extract_unique_files(nodes: List[Dict]) -> List[str]`
从Source/Sink节点中提取唯一的文件路径。

```python
def _extract_unique_files(self, nodes: List[Dict]) -> List[str]:
    """从节点列表中提取唯一的文件路径。"""
    files = set()
    for node in nodes:
        file_path = node.get('file', '') or node.get('file_path', '')
        if file_path:
            files.add(file_path)
    return list(files)
```

#### 2. `_read_multiple_source_files(file_paths: List[str], max_files: int = 5) -> str`
批量读取源文件并格式化为易读文本。

**特性**：
- ✅ 支持批量读取（最多5个文件，防止过大）
- ✅ 单文件大小限制（最多5000行，防止超出LLM上下文）
- ✅ 错误处理（读取失败时提供友好提示）
- ✅ 格式化输出（带文件名标记和代码块）

```python
def _read_multiple_source_files(self, file_paths: List[str], max_files: int = 5) -> str:
    """读取多个源文件并格式化为易读的文本。"""
    # 限制文件数量
    file_paths = file_paths[:max_files]
    
    source_contents = []
    for file_path in file_paths:
        content = self._read_source_file(file_path)
        
        # 限制单个文件大小（最多5000行）
        lines = content.split('\n')
        if len(lines) > 5000:
            content = '\n'.join(lines[:5000]) + f"\n\n... (已截断，总共{len(lines)}行)"
        
        source_contents.append(f"""
### 文件: {file_path}
```
{content}
```
""")
    
    return "\n".join(source_contents)
```

## 🔧 工作流程对比

### 之前（只靠提示词）
```
1. CodeQL查询返回空
2. 执行断流点查询
3. 将CodeQL结果传给LLM
4. 提示词告诉LLM: "请读取源文件"
```

### 现在（代码主动控制）
```
1. CodeQL查询返回空
2. 执行断流点查询
3. 【新增】代码提取Source/Sink文件路径
4. 【新增】代码主动读取这些文件
5. 【新增】将源码内容格式化
6. 将CodeQL结果 + 源码内容传给LLM
7. LLM分析（已经有完整源码）✅
8. 生成精确的修复条件
```

## 🛡️ 安全和限制

### 文件数量限制
```python
max_files: int = 5  # 最多读取5个文件
```

**原因**：防止读取过多文件导致LLM上下文溢出

### 单文件大小限制
```python
if len(lines) > 5000:
    content = '\n'.join(lines[:5000]) + "... (已截断)"
```

**原因**：大型文件（如auto-generated）会占用过多token

### 总大小估算
```
5个文件 × 5000行 × 80字符/行 ≈ 2M字符 ≈ 500K tokens
```

当前LLM上下文窗口通常是100K-200K tokens，因此：
- ✅ 通常情况（2-3个小文件）：完全够用
- ⚠️ 极端情况（5个大文件）：可能接近上限
- 💡 未来优化：可以根据LLM上下文大小动态调整
